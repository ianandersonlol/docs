## What is a Subagent?

A **subagent** is a separate agent instance that your main agent spawns to handle a specific task. The main agent delegates work, the subagent goes off and does it, and the results come back. It's like asking a colleague to handle one part of a project while you focus on another.

Subagents are useful for three reasons:

1. **Parallelization** — Multiple subagents run simultaneously, so independent tasks complete in the time it takes for the slowest one, not the sum of all of them
2. **Context protection** — Subagent work doesn't pollute your main conversation with intermediate noise
3. **Cost optimization** — You can run subagents on cheaper models (like Haiku) for tasks that don't need the full power of Opus or Sonnet

## Claude Already Spawns Subagents

Here's something the rest of this page might obscure: **Claude Code already spawns subagents on its own, without any configuration from you.** When Claude determines that a task would benefit from parallelization or context isolation, it creates subagents dynamically based on the conversation context. You don't need to define anything in `.claude/agents/` for this to work.

For example, if you ask Claude to search for a function definition across your codebase while also looking up documentation for a library, it may spawn an exploration subagent for the codebase search while handling the documentation lookup itself. You didn't configure this — Claude read the situation and decided parallelization made sense.

This built-in behavior is surprisingly good. Claude uses your conversation context to decide what each subagent should do, what tools it needs, and how to synthesize the results. For many workflows, this is all you need.

## When to Define Custom Subagents

So if Claude already handles subagent spawning on its own, why would you ever define custom agents in `.claude/agents/`? Three reasons:

1. **Cost control** — You want to force a specific cheaper model. When Claude spawns subagents dynamically, it uses its own judgment about model selection. A custom agent definition lets you pin a task to Haiku explicitly, which matters if you're running the same expensive workflow repeatedly.

2. **Consistency** — You want the exact same behavior every time. Claude's dynamic subagents are shaped by whatever conversation context exists at the moment. If you need a literature search that always checks the same sources in the same order and outputs in the same format, a custom agent definition gives you that reproducibility. You're not relying on Claude to reinvent the approach each time.

3. **Explicit tool restrictions** — You want to lock down what the subagent can access. A custom agent definition with a specific `tools` list ensures the subagent can't accidentally write files when it should only be reading, or can't access the web when it should only be working locally.

If none of those apply — if you're fine with Claude deciding how to parallelize, what model to use, and what tools to give the subagent — you probably don't need custom agent definitions. Just describe what you want and let Claude figure out the orchestration.

## When NOT to Use Subagents (Custom or Otherwise)

Whether Claude spawns them automatically or you define them explicitly, subagents are the wrong choice when:
- Tasks are sequential and each depends on the previous result — just do them in order
- The task is simple enough for your main agent — spawning a subagent has overhead
- **The task's output is something you want to think about** — this is the big one. More on this below.

## Good Use vs. Bad Use of Subagents

This is important enough to get its own section, because getting this wrong is one of the most common mistakes.

### Good: Research Subagent

```markdown
<!-- .claude/agents/research.md -->
---
name: research
description: Research best practices and implementations for unfamiliar topics
model: haiku
tools:
  - WebSearch
  - WebFetch
  - Read
---

# Research Agent

When the main agent encounters a feature or technique it's unsure about:

1. Search GitHub for well-regarded implementations
2. Read relevant documentation and tutorials
3. Check for recent papers or blog posts on best practices
4. Summarize findings with:
   - Recommended approach and why
   - Common pitfalls to avoid
   - Links to reference implementations
```

**Why this is a good subagent:** The research process generates a lot of intermediate context (web pages, READMEs, documentation) that your main agent doesn't need to see. You just need the summary. Plus, Haiku can handle web search and summarization just fine — no need to pay for Opus to read Stack Overflow. The research results get distilled into a concise summary that comes back to your main context.

### Bad: Sequence Similarity Network Analysis

Imagine a subagent that:
1. Runs an all-vs-all BLAST
2. Builds a sequence similarity network
3. Clusters the network
4. Analyzes the clusters
5. Selects candidate proteins to order

**Why this is a bad subagent:** Steps 1-3 are mechanical — running commands, parsing output. But step 4 (analyzing the clusters) and step 5 (selecting candidates) are *the science*. That's the part where your judgment matters. If you farm this out to a subagent, you lose visibility into the analysis, and you can't ask follow-up questions about the results because the full context is trapped in the subagent's conversation.

**What to do instead:** Make this a **skill**. A skill that teaches your main agent how to run the BLAST, build the network, and cluster it. The commands themselves are relatively token-light — Claude doesn't need to read the raw BLAST output because you can use `!bash blastp ...` to run it and only the output gets passed to context. Then the analysis happens in your main conversation where you can see it, question it, and redirect it.

<Note>
  **Security note on `!bash`:** The `!bash` prefix runs a command and passes only its output to context (not the command itself being "read" by the agent). This is powerful for keeping context clean, but it's also an attack vector. If you're using skills written by other people, inspect them carefully — a malicious skill could use `!bash` to execute arbitrary commands. Always review skills before adding them to your project.
</Note>

The rule of thumb: **if the output of a task is something you want to think about, keep it in your main conversation.** If the output is just a summary you'll glance at, a subagent is fine.

## Defining Custom Subagents

When you do need the consistency, cost control, or tool restrictions that justify a custom agent, here's how to set them up.

<Tabs>
  <Tab title="Claude Code">
    In Claude Code, you define custom subagents in `.claude/agents/`:

    ```markdown
    <!-- .claude/agents/literature-search.md -->
    ---
    name: literature-search
    description: Search scientific literature and summarize findings
    model: haiku
    tools:
      - WebSearch
      - WebFetch
      - Read
      - Write
    ---

    # Literature Search Agent

    You are a scientific literature research assistant. When given a topic:

    1. Search for recent papers (prioritize last 3 years)
    2. Focus on high-impact journals and preprint servers
    3. For each relevant paper found:
       - Title, authors, journal, year
       - One-sentence summary of the key finding
       - Relevance to the query
    4. Synthesize findings into a brief overview
    5. Suggest 2-3 follow-up questions or related searches

    ## Search strategy
    - Start broad, then narrow based on initial results
    - Check both PubMed and Google Scholar
    - Include preprints from bioRxiv/arXiv if relevant
    - Note if a finding is from a preprint vs. peer-reviewed
    ```

    Claude Code will use this agent when it determines literature search is needed, or you can request it explicitly:

    ```
    > Use the literature-search agent to find recent papers on de novo enzyme design using diffusion models
    ```

    Note the `model: haiku` in the frontmatter — this runs the subagent on a cheaper model since literature search and summarization don't need the most powerful model.
  </Tab>
  <Tab title="Codex">
    Codex CLI supports explicit subagent spawning with three built-in agent types: **default**, **worker** (execution-focused), and **explorer** (read-heavy analysis). You can also define custom agents as TOML files in `~/.codex/agents/` (personal) or `.codex/agents/` (project-scoped).

    Subagents run in parallel with configurable limits — `max_threads` (default 6) controls how many run simultaneously, and `max_depth` (default 1) prevents recursive delegation. You manage active subagents with the `/agent` slash command.

    The desktop app makes this especially visual — you can see all your agent threads running in parallel with their own terminals and worktrees. For cloud-based async work, `codex cloud` lets you fire off tasks in isolated containers and come back later.
  </Tab>
  <Tab title="Gemini CLI">
    Gemini CLI supports spawning sub-tasks through its orchestration features. The configuration is less granular than Claude Code's agent definitions.
  </Tab>
  <Tab title="Cursor">
    Cursor doesn't have a native subagent concept. It's fundamentally a single-agent experience embedded in an editor.
  </Tab>
  <Tab title="OpenClaw">
    <Warning>
      OpenClaw users don't need subagents — they already have 36 of them and $200/day in API costs to show for it.
    </Warning>
  </Tab>
</Tabs>

## Science-Relevant Examples

### Data Validation Agent

```markdown
<!-- .claude/agents/data-validator.md -->
---
name: data-validator
description: Validate experimental data files for common issues
model: haiku
tools:
  - Read
  - Bash
  - Glob
---

# Data Validation Agent

When given data files to validate:

## For CSV/TSV files:
- Check for consistent column counts across rows
- Identify missing values and their frequency
- Flag outliers (>3 standard deviations from mean for numeric columns)
- Check for duplicate rows
- Verify expected column headers are present

## For FASTA files:
- Verify format (> header lines, sequence lines)
- Check for non-standard characters
- Report sequence count and length distribution
- Flag duplicate sequence IDs

## For PDB files:
- Check for missing atoms in residues
- Identify chain breaks
- Report resolution and R-factors
- Flag unusual B-factors

## Output
Write a validation report to `validation_report.md` with:
- Summary: pass/fail with issue count
- Details: each issue with line numbers
- Recommendations: suggested fixes
```

### Usage: Combining Multiple Agents

Here's what a real workflow looks like — you're designing a new enzyme variant and want to parallelize the analysis:

```
> I just finished running AlphaFold on 50 TEM-1 variants. The predictions are
> in ./af_predictions/. I need three things done:
>
> 1. Validate all the output files for completeness
> 2. Analyze the structure predictions and rank by confidence
> 3. Search for recent papers on TEM-1 omega loop mutations
>
> Run these in parallel using the appropriate agents.
```

Claude Code will:
- Spawn `data-validator` (on Haiku) to check the AlphaFold outputs
- Spawn `structure-analyzer` to rank predictions by confidence
- Spawn `literature-search` (on Haiku) to find omega loop mutation papers
- Coordinate the results and present a unified summary

All three run simultaneously. What would take you 30+ minutes of sequential work happens in the time it takes for the slowest task to complete. And your main context stays clean — you get summaries, not the raw intermediate output of 50 structure analyses.

## Best Practices

<AccordionGroup>
  <Accordion title="Use cheaper models where you can">
    Not every subagent needs Opus. Literature search, data validation, file parsing — Haiku handles these fine at a fraction of the cost. Save the expensive models for tasks that require deep reasoning. Set the model in the agent's frontmatter with `model: haiku`.
  </Accordion>

  <Accordion title="Define clear boundaries">
    Each agent should have a specific, well-defined job. "Analyze stuff" is bad. "Validate FASTA files for format and content issues" is good. Clear boundaries prevent agents from stepping on each other's work.
  </Accordion>

  <Accordion title="Specify tools explicitly">
    List exactly which tools each agent can use. A literature search agent doesn't need file write access. A data validator doesn't need web search. Minimal permissions = fewer surprises.
  </Accordion>

  <Accordion title="Define output format">
    Every agent should know where and how to write its results. "Write a summary to validation_report.md" is better than "tell me what you found." Structured output is easier for your main agent to synthesize.
  </Accordion>

  <Accordion title="Don't over-parallelize">
    Three parallel agents for genuinely independent tasks? Great. Fifteen agents for a task that could be done sequentially? You're OpenClaw-brained. Parallelism has overhead — spawn agents only when the tasks are truly independent and substantial enough to justify it.
  </Accordion>

  <Accordion title="Keep science in your main context">
    This is the most important one. If the subagent is doing work whose results you need to *think about* — analyzing data, interpreting results, making scientific decisions — it probably shouldn't be a subagent. Keep that in your main conversation where you can engage with it.
  </Accordion>
</AccordionGroup>
