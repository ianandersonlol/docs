title: Choosing the Right Tool
description: When to use a skill vs. a slash command vs. a subagent vs. just talking to Claude — or nothing at all
---

You now know about skills, slash commands, subagents, and MCP. The natural question: **when do I use which?**

Here's the honest answer: most of the time, you should just talk to your agent directly. The customization tools exist for when you hit a pattern you're repeating, not for over-engineering your setup on day one.

## The Decision Guide

### "I need to do something once"

**Just type it out.**

Seriously. The most underrated agentic AI technique is just describing what you want in plain English. No skill file, no slash command, no configuration. Just:

```
> Parse this PDB file and tell me about the binding site residues
```

Claude Code will figure it out. If it doesn't do it exactly the way you want, tell it. Iterate. This is the fastest path to getting something done.

**Use this when:** it's a one-off task, you're exploring, or you haven't done this enough times to know your preferred format.

### "I keep asking for the same thing"

**Make a slash command.**

If you've asked Claude to "analyze this sequence and give me MW, pI, and charge at pH 7" three times this week, that's a pattern. Save it:

```markdown
<!-- .claude/commands/analyze-seq.md -->
Analyze the following sequence: $ARGUMENTS
Report: type, length, MW (kDa), pI, charge at pH 7, notable features.
Be concise. One-line summary first.
```

Now you type `/analyze-seq MVLSPADKTN...` and get consistent results every time.

**Use this when:** you have a prompt you use repeatedly, it's relatively short, and you want consistent output format.

### "This task requires specialized knowledge"

**Write a skill.**

If the task involves domain-specific conventions, multi-step workflows, or knowledge about your environment that Claude wouldn't know by default, that's a skill:

```markdown
<!-- .claude/skills/local-blast.md -->
When running BLAST: use the conda blast_env, databases are at ~/blast_db/,
default e-value 1e-5, always use outfmt 6 for parsing...
```

Skills are richer than slash commands — they encode *how* to do something, not just *what* to do.

**Use this when:** the task requires environment-specific knowledge, has multiple steps that should be done in a specific order, or involves conventions that would be tedious to re-explain.

### "I need to connect to an external system"

**Maybe set up an MCP server. But probably just use a skill.**

As discussed in the MCP section, for bioinformatics specifically, skills and direct CLI access almost always get you further than MCP. Reserve MCP for genuinely external systems like databases or institutional APIs.

**Use this when:** you need persistent connections to external systems — databases, APIs, internal services — and the tool doesn't have a CLI.

### "I have genuinely parallel work"

**Spawn subagents.**

If you have multiple independent tasks — validate data, analyze structures, search literature — and they don't depend on each other, run them in parallel:

```
> Run these in parallel:
> 1. Validate the AlphaFold outputs in ./predictions/
> 2. Search for papers on TEM-1 omega loop stability
> 3. Check if my SLURM jobs from yesterday completed successfully
```

**Use this when:** you have 2+ independent tasks that would take meaningful time if done sequentially, and the results don't need your deep engagement.

### "I do this all the time and it never changes"

**Write a script.**

This is the one people forget. If a task is fully repeatable — same inputs, same steps, same outputs — and doesn't require any judgment or setup from the agent, it's a better use of tokens to have Claude write you a Python script or bash script and then just *run the script*. You don't need an agent to do something a cron job could handle.

<Warning>
  Tokens are currently **heavily subsidized** by model makers. The prices you pay today are a fraction of the actual cost. This will not last forever, and it will end sooner than most people realize. Build habits now that are token-efficient. If something can be a script instead of an agent interaction, make it a script.
</Warning>

**Use this when:** the task is deterministic, repeatable, and doesn't benefit from AI judgment.

## Quick Reference

| Situation | Tool | Effort to set up |
|-----------|------|-----------------|
| One-off task | Just type it | None |
| Repeated prompt | Slash command | 2 minutes |
| Complex domain task | Skill | 10-15 minutes |
| Fully repeatable task | A script | 5-30 minutes |
| External system access | MCP server (maybe) | 30 min - few hours |
| Parallel independent tasks | Subagents | 10-15 min per agent |

## Common Mistakes

<AccordionGroup>
  <Accordion title="Over-engineering too early">
    Don't write skills and slash commands on day one. Use Claude Code for a week first. See what patterns emerge. Then codify the ones you actually repeat. Premature optimization applies to agent configuration too.
  </Accordion>

  <Accordion title="Making everything a subagent">
    Not every task needs to be parallelized. If tasks are sequential (step 2 needs the output of step 1), subagents add overhead for no benefit. And if the output is something you need to think about scientifically, keep it in your main conversation.
  </Accordion>

  <Accordion title="Using an agent when a script would do">
    If you're running the same BLAST search with the same parameters on a new sequence every day, that's not an agent task — that's a bash one-liner. Have Claude write the script once, then run it yourself. Save the agent for tasks that actually require intelligence.
  </Accordion>

  <Accordion title="Putting environment details in slash commands instead of skills">
    A slash command says "do X." A skill says "here's how to do X in our environment." If your slash command is growing past 10 lines because you're explaining where databases live and what conda environments to use, it should be a skill.
  </Accordion>

  <Accordion title="Ignoring CLAUDE.md">
    Before you write any skills or commands, put your project-wide conventions in `CLAUDE.md`. Things like "always use mamba instead of conda" or "our lab uses one-letter amino acid codes" belong there, not duplicated across every skill.
  </Accordion>

  <Accordion title="The OpenClaw trap">
    The temptation to build an elaborate multi-agent system with specialized agents for everything is real. Resist it. One good agent with well-defined skills will outperform an army of poorly-configured agents every time — and cost a fraction of the tokens.
  </Accordion>
</AccordionGroup>

## A Note on AI Agents and Science

I want to end with something that I think is important and that gets lost in the hype.

For science specifically, I think it is better to treat these agentic systems as a **collaborator** you're working alongside, not a workforce you're managing. The human-in-the-loop component is really important. Keep the interaction conversational.

It's easy — and tempting — to offload more and more of your workflow to skills and subagents that just run automatically. That's great in industry, and great for other fields, but in science it might not be the best approach. **Your value as a scientist is your ability to do science** — to think critically, to notice patterns, to ask the right questions, to solve hard problems. Using AI agents is easy to learn. Work on a side project over a weekend and you can be close to an expert, or at least enough of one that your job prospects are safe. Being a good scientist is the differentiator. Keep doing that.

There is a lot of hype right now around the idea that this is a step change from chatbots, and that you shouldn't be treating them as a chatbot. I think that's the wrong way to look at it. Chatbots have been agentic for a long time now, we're just now noticing. Sure, minimize how much you have to explain to your agent — use skills, use CLAUDE.md, set up your environment. But always keep the human in the loop. The agent handles the doing. You handle the thinking. That's the partnership that works.

## The Bottom Line

Start simple. Type what you want. When you notice a pattern, save it as a slash command. When you need domain expertise, write a skill. When you have parallel work, use subagents. When something is fully repeatable, write a script.

That's it. No souls, no heartbeats, no identity files. Just a capable agent and a clear way to extend it when you need to.
