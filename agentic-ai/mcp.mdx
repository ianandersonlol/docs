## What is MCP?

**Model Context Protocol (MCP)** is an open standard that lets AI agents connect to external tools and data sources through a unified interface. Think of it as **USB for AI** — instead of every agent needing custom integrations for every tool, MCP provides a standard plug that works everywhere.

Before MCP, if you wanted Claude Code to interact with your database, you'd need to write a custom tool or script. With MCP, someone writes a database server once, and any MCP-compatible agent can use it.

## How It Works

MCP follows a client-server architecture:

1. **MCP Servers** expose capabilities — tools, resources, prompts — through a standard protocol
2. **MCP Clients** (like Claude Code, Claude Desktop) connect to servers and make those capabilities available to the LLM
3. The LLM decides when and how to use the available tools, just like any other agent tool

```
You → Agent (Claude Code) → MCP Client → MCP Server → External Tool
                                                          ↓
You ← Agent (Claude Code) ← MCP Client ← MCP Server ← Results
```

## Setting Up MCP in Claude Code

MCP servers are configured in your Claude Code settings. You can set them up at the project level or globally:

```json
// .claude/settings.json (project-level)
{
  "mcpServers": {
    "filesystem": {
      "command": "npx",
      "args": ["-y", "@anthropic/mcp-filesystem-server", "/path/to/allowed/directory"]
    },
    "postgres": {
      "command": "npx",
      "args": ["-y", "@anthropic/mcp-postgres-server", "postgresql://localhost/mydb"]
    }
  }
}
```

Once configured, Claude Code automatically discovers the available tools and can use them during your conversation.

## My Honest Take: MCP and Bioinformatics

MCP has a lot of promise. In other fields — web development, enterprise software, DevOps — it's genuinely useful for connecting agents to databases, APIs, and cloud services. The standard is well-designed and the ecosystem is growing.

**For bioinformatics, I don't think MCPs are remotely useful.** (Hehe, pun intended.)

Here's why: most bioinformatics tools are command-line programs. AlphaFold, BLAST, GROMACS, samtools, bedtools — you run them from a terminal. There's an AlphaFold 2 MCP out there. AlphaFold 2 is *run via the command line*. Why do you need an MCP server wrapping a CLI tool when your agent already has access to the command line? It's an extra layer of abstraction that adds complexity without adding capability.

Even if you're using cloud compute for heavy jobs, there are better ways to implement that than an MCP server. A skill that knows how to submit SLURM jobs, or a simple wrapper script, will get you further with less overhead.

In my experience, **skills and CLIs are almost always more useful than MCPs** for science workflows. A skill that teaches Claude how to run BLAST is simpler, more transparent, and easier to debug than an MCP server that wraps BLAST. The skill lives in a markdown file you can read and edit. The MCP server is a running process that adds a network layer between your agent and your tool.

<Warning>
  Be very wary of MCP servers in bioinformatics. Before installing one, ask yourself: "Could I just teach my agent to run the command directly?" If the answer is yes — and it almost always is — skip the MCP and write a skill instead.
</Warning>

### Where MCPs *might* make sense in science

To be fair, there are a few scenarios where MCP could add value:

- **Lab databases** — If your lab has a PostgreSQL or SQLite database storing experimental results, an MCP server gives your agent structured access to it
- **Internal APIs** — If your institution has REST APIs for instrument data, sample tracking, etc.
- **Persistent services** — Things that genuinely need a running server, like a local language model or a monitoring dashboard

But for the 90% case of "I want my agent to run a bioinformatics tool" — just use the command line. That's what it's there for.

## MCP vs. Skills vs. Slash Commands

| Mechanism | What it does | When to use |
|-----------|-------------|-------------|
| **MCP** | Connects agents to external tools/data via a standard protocol | When you need to interface with a genuinely external system (database, API, running service) |
| **Skills** | Teach the agent reusable behaviors and knowledge | When you want the agent to know *how* to do something specific — this is usually what you want in bioinfo |
| **Slash Commands** | Quick prompt templates | When you have a common prompt you want to reuse |
